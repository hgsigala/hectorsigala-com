The big idea

You tell it what to build.
It writes the code.

Claude Code is a tool that lives in your computer's terminal. You talk to it in plain English — like texting a friend who happens to be a really fast coder — and it builds whatever you ask.

You say

"Make a website with 4 buttons. Each one plays a different fart noise when I click it. Black background, big red buttons."

It does

Creates a folder, writes the HTML, writes the CSS, wires up the JavaScript, finds 4 fart sounds, and opens the page in your browser. About 30 seconds.

Who does what

You're the boss.
It's the worker.

Claude Code can write code faster than anyone on Earth. But it doesn't know what you want. That's your job.

You decide what to build

Pick the idea. The weirder and more "yours," the better. Nobody else can do this part.

It writes the code

Like having a really good coder who never gets tired and never says "that's a dumb idea."

You test it

Open the page. Click around. Tell it what's wrong. Don't trust it when it says "it works" — go check.

Three rules. Always.

Don't skip these. Ever.

1

Don't type real personal info into Claude

No home address, no phone numbers, no passwords, no school logins, no real credit card numbers. If you need a fake one to test something, ask Dad and we'll make one up together.

2

If something feels weird, tell Dad right away

If a website you built shows something scary, if Claude Code asks you to do something that feels off, or you just have a bad feeling — come find me. No project is more important than that.

3

Show family first before sharing anywhere

The things you build are yours. Don't post them on YouTube, Discord, TikTok, Reddit, or send them to strangers. Show your family first, and we'll decide together if and how to share.

How to make one

Just tell Claude Code:

"Make a new project in ~/Sites/gael/ called all-about-me."

It creates the folder, sets up the files, and tells you where everything is. No need to know what any of it does yet.

The skill

How to talk to Claude Code

More detail = better result. Describe what you want like you'd describe it to a friend who can't see what's in your head.

Vague — you'll be unhappy
"make me a game"

It will make a game. Probably not the one you want.

Specific — you'll be happy
"Make a clicker game. The button is a giant Pikachu. Every click adds 1 to a counter. After 50 clicks, the Pikachu turns into a Raichu. Yellow background."

It builds exactly what you imagined.

Tip: if it builds the wrong thing, that means you didn't describe it enough. Don't get mad — add more detail and ask again. "Same thing, but bigger buttons and make the counter green."

It will happen

When it gets stuck

Claude Code makes mistakes. So does every other coder. The trick isn't avoiding mistakes — it's knowing what to do when they happen.

1. Read the error out loud

Even if half of it looks like alien language, the part you understand is usually enough to know what went wrong.

2. Tell Claude what happened

"That didn't work. The page is blank." or "The button doesn't do anything when I click it." Be specific.

3. Don't trust "it works"

Claude will sometimes say it's done when it isn't. Open the page yourself and click everything. Trust your eyes.

4. Still stuck? Ask Dad

Some bugs need an adult. No shame — even pro coders ask for help every day. That's the job.

Looking ahead

What should I build next?

When you're picking your next project, run through these questions. The answer usually shows up.

  • What's something that bugs you every day?
  • What's a website you visit a lot — could you build a worse-but-yours version?
  • What's the simplest game you can think of? Could you make it?
  • What's something you wish you had a button for?
  • What's a chore you'd love to automate?
  • If a computer could build literally anything for you in 10 minutes, what would you pick?
  • (Ivan) What's a tool your friends would actually use? Build that.

The best projects come from real annoyances in your real life. Not the cool-sounding thing — the actual thing.

Get it on the real internet

Once your project is good, push it to GitHub. Then connect it to Vercel. Now it lives on a real URL you can open on your phone.

GitHub = save button for code

A cloud folder for your project. Free. Every change is saved forever. If your laptop dies, your code is safe.

Ask Claude Code: "Put this project on my GitHub at hgsigala — no wait, my GitHub — and push it."

Vercel = "put it online"

Vercel takes a GitHub repo and turns it into a real website at your-project.vercel.app. Free for personal stuff.

Dad will walk you through connecting Vercel the first time. After that, every time you push to GitHub, your site updates automatically.

Don't deploy week 1. Build the project. Use it locally. Make sure you actually like it. Then put it online.
